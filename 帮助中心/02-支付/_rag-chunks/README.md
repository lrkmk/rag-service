# 支付大类 — RAG 切片

对「02-支付」下 3 篇源文章做的切片，套用「04-售后票务/退票」示例验证过的切法。`支付.md` 是导航/链接列表，不含实质内容，未纳入切片。

## 文件

- `parents.jsonl` — 文章级 parent 记录（3 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（15 条）

## 切分依据

1. **文章为最小一级边界**：3 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child（多币种指南 7 个、汇率使用说明 4 个、混合支付指南 4 个）。
2. **规则单元整段保留**：不按字数/token 数机械切断。例如 `dbz-c07`（API多币种询价与结算）把"currency字段非必填时的默认取值逻辑"和"需在询价阶段指定、其余接口参数不变"两条耦合规则放在同一 chunk，未从中间切开；`hl-c02`/`hl-c03` 把"报价支付阶段汇差由Atlas承担" vs "退款差额结算阶段按处理时点汇率折算"两条对比规则拆成两个独立 chunk，因为二者是可独立检索、不互相依赖的规则单元。
3. **航司特例单独打标签**：本大类三篇文章内容均与具体航司无关（多币种、汇率、支付方式均为平台通用规则），`applicable_carrier` 统一为 `通用`。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 的一句话摘要，帮助模型判断这条规则出自"多币种"“汇率”还是"混合支付"（三者同属 02-支付 一级分类，容易被模糊检索混淆，例如"汇率波动"既可能问汇率使用说明，也可能问多币种账户折算）。
5. **FAQ 单独处理**：本次示例未覆盖 FAQ（10-常见问题），原则与 04-售后票务 一致——按一问一答切，不按字数切。
6. **Metadata 字段**：每条 child 含 `level1_category`（支付）、`level2_category`（多币种/汇率/混合支付，按具体文章主题区分，而非固定值）、`applicable_carrier`、`updated_at`（源文章及站点导航文件均未标注更新日期，故三篇统一记为 null）、`source_path`（可追溯到本地归档原文件，如 `02-支付/多币种指南.md`）。
7. **数字类规则抽表**：本大类无需抽表的大型结构化数据表——多币种指南中的"支持货币列表"仅 24 项且为静态清单（非按条件变化的数值规则），直接保留在 `dbz-c02` 一个 chunk 内即可语义检索，未做额外结构化拆分。

## 示例：一条完整 child 记录

```json
{
  "chunk_id": "hl-c02",
  "parent_id": "huilv-shiyong-shuoming",
  "section": "二、关于汇率波动对金额的影响——报价与支付阶段",
  "rule_type": "计算规则",
  "applicable_carrier": "通用",
  "updated_at": null,
  "text": "报价与支付属于同一交易确认流程，客户在当前报价下完成支付时，Atlas将以报价展示金额作为客户应付金额；报价至支付期间如因短时间汇率波动产生汇差，该部分汇差风险由Atlas承担，不会再向客户调整支付金额。"
}
```
