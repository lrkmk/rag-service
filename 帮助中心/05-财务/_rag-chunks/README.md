# 财务大类 — RAG 切片

对「05-财务」大类下 2 篇源文章做的切片，沿用「04-售后票务/退票」示例的同一套规则（`财务.md` 是导航链接列表，非正文内容，已跳过）。

## 文件

- `parents.jsonl` — 文章级 parent 记录（2 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（9 条）

## 切分依据

1. **文章为最小一级边界**：2 篇文章互不跨界拼接，各自独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：不按字数/token 机械切断。例如 `rmb-c05`（首笔充值）把“10–20分钟到账”“到账后账户开通、后续自动入账”“未到账联系客户经理排查”这组条件+结论保持在同一个 chunk 内；`cz-c02`（充值金额计算规则）把“应为扣除银行手续费后的转账金额”这条规则连同示例数字（20000英镑－15英镑＝19985英镑）整段保留，不拆分公式与例子。
3. **航司特例单独打标签**：本大类内容均与具体航司无关（充值/财务操作面向客户主体而非航司），因此 `applicable_carrier` 统一为 `通用`。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中后可回带 parent 摘要，帮助模型区分「人民币自动充值操作指南」（面向中国主体、易宝渠道、开户流程）与「充值用户指南」（ATRIP/Flight Deck 上提交充值请求与查历史的通用操作，两篇标题相近、场景易混淆）。
5. **FAQ 单独处理**：本大类无 FAQ 源文件，未涉及。
6. **Metadata 字段**：每条 child 含 `level1_category`（财务）、`level2_category`（人民币充值 / 账户充值，按各自文章主题区分）、`applicable_carrier`、`updated_at`（两篇源文章均无版本/更新日期标注，且导航文件 `财务.md` 也未列出更新时间，故记为 `null`）、`source_path`（可追溯到本地归档原文件）。
7. **数字类规则未做结构化抽表**：本次涉及的数字（10–20分钟到账时限、20000英镑/15英镑充值金额示例）体量小、单一场景，直接保留在对应 child 的文字说明中，未采用类似退票时限表的结构化 JSON 抽取。

## 示例：一条完整 child 记录

```json
{
  "chunk_id": "cz-c02",
  "parent_id": "chongzhi-yonghu-zhinan",
  "section": "充值金额填写说明",
  "rule_type": "计算规则",
  "applicable_carrier": "通用",
  "updated_at": null,
  "text": "充值金额计算规则：输入的充值金额应为从客户银行扣除的银行收费后的转账金额。例如：转账金额20000英镑，客户的银行收费15英镑，充值金额=20000英镑-15英镑=19985英镑。"
}
```

## 判断说明

- `人民币自动充值操作指南.md` 明确面向「中国主体客户」，通过「易宝」完成人民币充值；`充值用户指南.md` 的截图路径为「Flight Deck」而查询路径中用的是「ATRIP」，两者应为同一系统的不同称呼/版本，内容判定为通用充值操作流程，未标注特定币种限制，故未强行归为同一 `level2_category`，而是按各自文章的实际主题分别标注为「人民币充值」与「账户充值」。

## 未覆盖范围

这只是「财务」大类目前仅有的 2 篇正文文章的示范。若后续财务大类下新增文章（如账单、发票、对账等子类），可套用同样的切法。
