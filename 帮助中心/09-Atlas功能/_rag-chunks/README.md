## 切分依据

1. **文章为最小一级边界**：3 篇文章互不跨界拼接，各自独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：按源文档的条款编号/小标题切分。例如 `af-c12`（购票信息不符）把"错过航班需在原航班起飞后4小时内通知"这一时限条件与其后两条互斥补救措施（1.1 同舱位机票+最高150美元报销 / 1.2 全额退款+额外30美元赔偿）作为一个不可再分的条件+结论组合保留在同一 chunk；`af-c05`（访问密钥续期规则）把有效期、推荐轮换周期（3个月）、可选周期（3/6/12个月）、并存上限（2个）、到期提醒、失效后果、恢复流程作为一条完整的时限规则链条整体保留，未按小分句拆开。
3. **航司特例**：本分类 3 篇文章均为Atlas平台通用条款，不涉及具体航司差异，`applicable_carrier` 统一为"通用"。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，可回带 parent 摘要帮助判断规则出自哪篇文章（尤其"票务承诺"与"售后票务/退票"分类容易在语义上混淆，需要靠 parent 摘要区分二者是"购票环节的平台赔付承诺"还是"票务变更后的退款规则"）。
5. **FAQ**：本分类源文件均为条款/说明文档，无 FAQ 形式内容，未涉及。
6. **Metadata 字段**：含 `level1_category`（Atlas功能）、`level2_category`（按文章区分）、`applicable_carrier`（均为"通用"）、`updated_at`、`source_path`。
7. **数字类规则未抽表**：本分类数字条款（如4小时/24小时通知时限、150美元/30美元赔偿额度、2年数据留存期）数量少且各自绑定特定业务场景，未构成可查表的结构化数据集，因此保留在对应文字 chunk 中。

## 判断记录

- **`updated_at` 全部为 `null`**：导航文件 `Atlas功能.md` 只列出一篇子文章（Atlas邮件服务）且未标注"最后更新日期"，无法确认其余两篇文章的更新日期来源，因此本分类所有记录的 `updated_at` 均记为 `null`。
- **语言处理**：`Atlas Security and Compliance.md` 源文件为纯英文服务条款，无中文对照（不同于 `Atlas票务承诺.md` 的英中双语格式）。为与本地文档站的中文语料保持一致，其 child 的 `text` 字段做了**忠实中文改写**（非逐字直译），所有数字/期限/数量（如"每3个月轮换""最多2个有效AK""到期后API报错""3/6/12个月可选周期"）均按源文一一对应保留，未增删事实。`Atlas票务承诺.md`、`Atlas邮件服务.md` 本身已是中文（或中英对照），直接以中文部分为准摘录。
- **与 `08-安全与合规/安全与合规说明.md` 的重叠评估（重点判断）**：两篇文章覆盖同一批安全能力——访问密钥（AK/SK）、SSO、IP白名单、MFA、密码策略、审核日志——但**角度互补，非完全重复**：
  - `08/安全与合规说明.md` 是**操作导览**：含具体 UI 路径（如"Flight Deck --> My Profile --> Authentication --> Password"）、按钮名称、截图步骤，以及只有它才给出的具体默认参数值（密码长度默认12/范围8-16、过期默认90天、锁定默认5次失败/30分钟、审核日志仅留存1年且单次下载最长1个月等）。
  - `09/Atlas Security and Compliance.md` 是**正式服务条款（ToS）**：只有它给出的是条款层面的补充规则——AK 建议每3个月轮换一次、同时最多2个有效AK、到期前邮件提醒管理员、密钥过期后API调用报错并停止交易、恢复访问需管理员重置并重新集成新密钥；同时用条款语言重申了"仅管理员可设置安全设置"等责任归属表述。
  - 结论：**保留为两条独立的 parent 记录**，未合并、未去重，因为两者在各自站点分类（08 vs 09）下服务不同意图——08 面向"怎么操作"，09 面向"条款约定了什么"，检索时应允许两者同时命中并互为补充。
