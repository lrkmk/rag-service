# 售前票务 — RAG 切片

对「03-售前票务」下 3 篇源文章做的切片，沿用「04-售后票务/退票」示例的同一套原则。`售前票务.md`（纯导航/链接列表，无实质内容）已跳过。

## 文件

- `parents.jsonl` — 文章级 parent 记录（3 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（12 条）

## 覆盖文章

1. **Atlas票务出票及取消规则说明** → `level2_category: 出票与取消规则`（4 个 child：出票时效标准 1 条 + 三类支付方式的取消触发条件各 1 条）
2. **如何停止出票？** → `level2_category: 停止出票`（3 个 child：ATRIP 操作、API 请求/响应格式、API 结果确认与状态码规则）
3. **如何重新生单？** → `level2_category: 重新生单`（5 个 child：功能说明、操作路径、步骤1-2+注意事项、步骤3-5、步骤6-7）

## 切分依据（沿用退票示例的 7 条原则）

1. **文章为最小一级边界**：3 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：不按字数/token 数机械切断。例如 `tz-c03` 把"8分钟后查询"与"orderStatus -3/2 两种结果含义"这组条件+结论耦合的规则保持在同一个 chunk 里；`cs-c03` 把"步骤1-2"与紧随其后的警示框（乘客数量限制、报错提示）合并为一条，因为警示内容直接约束这两步操作，拆开会丢失依赖关系。
3. **航司特例单独打标签**：本次 3 篇文章均为通用规则，未出现航司专属条款，`applicable_carrier` 统一为 `通用`。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中后可回带 parent 摘要，帮助模型判断规则出自"出票与取消""停止出票"还是"重新生单"（三者场景相邻、容易混淆，尤其是"停止出票"与订单取消规则的关系）。
5. **FAQ 单独处理**：本次 3 篇均为说明/操作类文章，无独立 FAQ 板块，不适用此条。
6. **Metadata 字段**：每条 child 含 `level1_category`（售前票务）、`level2_category`（各篇主题）、`applicable_carrier`、`updated_at`（源目录下无带更新日期的导航文件 `售前票务.md` 仅为链接列表，未标注日期，故全部记为 null）、`source_path`（可追溯到本地归档原文件）。
7. **数字类规则抽表**：本次内容体量小，未出现类似「131 家航司时限表」的大型结构化表格，故未产出额外的 `structured_data_ref` 文件；出票时效的两个百分比阈值（≥90%/5分钟、≥98%/1小时）直接保留在 `cq-c01` 文字中。

## 判断说明

- `如何停止出票？.md` 中 API 部分被拆为"请求/响应格式说明"（`功能说明`）与"结果确认规则"（`时限规则`）两个 chunk，因为前者是接口契约，后者是需要等待 8 分钟、依赖 `orderStatus` 取值的判定逻辑，二者检索意图不同。
- `如何重新生单？.md` 的 7 个操作步骤未逐条拆成 7 个 chunk，而是按语义就近合并为 3 组（步骤1-2 含前置条件校验、步骤3-5 为行李信息录入流程、步骤6-7 为确认与支付收尾），避免过细粒度导致检索时上下文碎片化。
