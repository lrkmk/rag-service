# 辅营服务子分类 — RAG 切片

对「04-售后票务/辅营服务」下 2 篇源文章（跳过纯导航文件 `辅营服务.md`）做的切片，沿用「退票」子分类示例的同一套规则。

## 文件

- `parents.jsonl` — 文章级 parent 记录（2 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（5 条）

## 切分依据

1. **文章为最小一级边界**：2 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：没有按字数/token 数机械切断。例如 `fy-c02`（暂不支持的辅营业务）把选座/餐食/特殊服务申请/其他未列明产品这一整份负面清单保持在同一个 chunk 里，不拆散；`xl-c02`（添加售后行李操作步骤）把四个操作步骤（含"提交后不可撤回"的风险提示）作为一个不可再分的操作流程整体保留，避免把警示语和触发它的步骤拆到不同 chunk。
3. **航司特例单独打标签**：本子类两篇文章均为通用规则（不区分航司），`applicable_carrier` 统一标为 `通用`。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 的一句话摘要，帮助模型判断这条规则出自「辅营服务」而非同一 04-售后票务 大类下的其他子类（如退票、改期）。
5. **FAQ/操作指南单独处理**：`如何添加售后行李？` 是一篇操作指南（非规则说明文），按"入口"与"操作步骤"两个自然单元切分，而不是按段落或字数切。
6. **Metadata 字段**：每条 child 含 `level1_category`（售后票务）、`level2_category`（辅营服务）、`applicable_carrier`、`updated_at`（本子类两篇源文章的导航列表 `辅营服务.md` 均未标注"最后更新日期"，故记为 null）、`source_path`（可追溯到本地归档原文件）。
7. **数字类规则抽表**：本子类源文章不含大批量结构化数值表格（如逐航司时限表），因此未产出配套的结构化 JSON 文件；若后续该子类新增此类表格内容，可参照「退票」子类的 `退票时限结构化表.json` 做法单独抽表。
