## 切分依据（对应 7 条原则）

1. **文章为最小一级边界**：3 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：没有按字数/token 数机械切断。例如 `fpz-c03`（废票窗口期设置规则）把"航司规定出票后24小时内可废票→Atlas窗口期设为出票后23小时内"与"航司规定仅出票当天可废票→Atlas窗口期设为出票当天23:30前"两组条件+结论耦合的规则放在同一个chunk，没有拆开；`fpz-c08`（执行失败即时通知）把"系统自动执行失败"这一触发条件与"5分钟内同步失败结果并结束流程"这一结论保持在同一段。
3. **航司特例单独打标签**：本子类三篇源文章均未包含航司专属条款（不同于「退票」子类下的FR/IJ两篇专属文章），所有child的`applicable_carrier`均为`通用`。若后续新增航司专属废票文档，应参照「退票」示例的FR/IJ做法单独打标签。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 的一句话摘要，帮助模型判断这条规则出自"废票"还是场景高度相似的兄弟子类"退票"（例如"7个自然日复核期"是退票的规则，"5分钟内失败通知"和"出票后23小时窗口期"才是废票的规则，容易混淆）。
5. **FAQ 单独处理**：`Atlas 废票服务快速指南.md` 中"三、常见问题快速查询"板块的问答实质上是"问题→指向对应文档"的索引形式（答案本身没有独立文字解答，而是链接到政策说明/服务条款文档），因此没有拆成8个独立child，而是合并为一条`fpk-c05`（FAQ/文档索引类型），检索命中时可直接告知用户该去查哪篇文档。
6. **Metadata 字段**：每条 child 含 `level1_category`（售后票务）、`level2_category`（废票）、`applicable_carrier`、`updated_at`、`source_path`（可追溯到本地归档原文件）。本子类三篇源文章及其所属的 `废票.md` 导航文件均未标注"最后更新日期"，因此所有记录的 `updated_at` 均为 `null`（与退票示例中缺失日期的处理方式一致）。
7. **数字类规则未做单独抽表**：本子类不涉及类似退票"131家航司差异化时限"的大规模结构化表格，废票窗口期只有"24小时内→23小时窗口"与"仅当天→23:30前"两条通用规则，数值直接保留在语义文本中（`fpz-c03`），无需额外结构化文件。

## 子类特有的判断取舍

- `Atlas 废票服务快速指南.md` 本质是一篇"文档导航/索引"文章（链接到本子类内的《Atlas废票政策说明》《如何申请废票？》，以及子类外的《Atlas 废票服务条款》《废票API文档》）。由于任务要求对它切片（不同于纯链接列表的`废票.md`），处理方式是：把每个"小节 + 它指向的文档在讲什么"作为一个child，`rule_type`标为`文档索引`或`操作入口`，不臆造原文没有的规则细节。其中链接的《Atlas 废票服务条款》和《废票 API 文档》不在本子类源文件范围内，`fpk-c02`中特别注明该条款文档归属"服务条款"分类，避免检索时误以为其内容也来自"废票"子类。
- `Atlas废票政策说明.md` 中"三、废票结果与退款说明"一节按自然段拆成两条child（`fpz-c09`结果不确定性 / `fpz-c10`结果判定依据及退款到账时间），因为二者分别对应"为何可能失败"与"退款到账取决于什么"两个不同角度的规则点，拆开更利于精确检索，且拆分点上没有把任何触发条件与其结论拆散。
- `如何申请废票？.md` 中步骤3的两个分支（"提交废票申请"与"提交退款申请"）操作路径完全不同（前者仅需确认，后者需额外填写履约日期+上传证明材料），因此拆成`fps-c05`、`fps-c06`两条，避免检索"如何提交退款申请"时被无关的"提交废票申请"步骤稀释。
