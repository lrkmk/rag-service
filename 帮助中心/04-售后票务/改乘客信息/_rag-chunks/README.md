# 改乘客信息子分类 — RAG 切片

对「04-售后票务/改乘客信息」下 2 篇源文章做的切片，沿用「退票」子分类的同一套切分规则（见 `../../退票/_rag-chunks/README.md` 中的 7 条建议）。`改乘客信息.md` 为纯导航/链接列表，不含实质内容，未纳入切片。

## 文件

- `parents.jsonl` — 文章级 parent 记录（2 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（14 条）

## 切分依据

1. **文章为最小一级边界**：2 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：没有按字数机械切断。例如 `gcz-c02`（特殊情况协助服务）把"提前24小时申请 / 不保证成功 / 收取服务费"三个并列条件放在同一个 chunk，因为三者共同构成同一受理前提，拆开会丢失"仅在满足全部条件时才协助"的语境；`gcz-c03`（改姓名场景）把"姓名更正 vs 姓名变更"的定义区分与该场景适用航司清单放在一起，因为清单只对"姓名更正"这一子情形生效，脱离定义单独存放清单会造成歧义。
3. **航司特例单独打标签**：`Atlas改乘客信息政策说明` 一文按"改姓名/改证件信息/改生日/改国籍/改性别"五个变更场景分别列出适用航司清单（VJ/IJ/ZE/A3/7C/FZ/3F 等），每个 child 的 `applicable_carrier` 字段精确记录该场景下的航司代码组合，检索"哪些航司支持改生日"之类问题时可直接命中对应 child，不与其他四个场景混淆。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 摘要，帮助模型判断这条规则出自"改乘客信息"（更正/更新身份信息）而非同大类下的"改期""退票"等易混淆兄弟子类。
5. **FAQ 单独处理**：本子分类下没有独立的 FAQ 源文件，未涉及此项。
6. **Metadata 字段**：每条 child 含 `level1_category`（售后票务）、`level2_category`（改乘客信息）、`applicable_carrier`、`updated_at`（源导航文件 `改乘客信息.md` 未标注任何"最后更新日期"，故本子类全部记为 `null`）、`source_path`（可追溯到本地归档原文件）。
7. **数字类规则抽表**：本子类涉及的航司清单/收费表规模较小（单表最多 6 行），不构成需要单独抽取结构化 JSON 的"大表"场景（对比退票子类 131 家航司的时限表），因此按原表格内容直接保留在 child 的 `text` 字段中，未做额外的 `structured_data_ref` 处理。
