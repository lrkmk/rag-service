# 航班及客票状态子分类 — RAG 切片

对「04-售后票务/航班及客票状态」下 5 篇源文章做的切片，沿用「退票」子类示例的同一套原则。跳过 `航班及客票状态.md`（仅为导航链接列表，无实质内容）。

## 文件

- `parents.jsonl` — 文章级 parent 记录（5 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（12 条）

## 切分依据

1. **文章为最小一级边界**：5 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：没有按字数/token 数机械切断。例如 `bhf-c02`（保护航班信息获取方式）把“使用 Atlas 邮箱服务”与“未使用 Atlas 邮箱服务”两条互斥路径保持在同一个 chunk 内，因为它们是同一判断下的两个分支，拆开会丢失对比关系；`pdhb-c03`（Flight Status 页面判断）把“状态=取消/状态=正常/查不到航班”三种条件-结论对作为一个不可再分的判定单元。
3. **本子类无航司特例**：5 篇源文章均为通用规则，`applicable_carrier` 统一标记为 `通用`，未出现类似「退票」子类中 FR/IJ 那样需要单独打标签的航司专属规则。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 的一句话摘要，帮助模型判断这条规则出自「航班及客票状态」而非同一 04-售后票务 大类下场景相近的「退票」「改期」等兄弟子类。
5. **多步骤操作按步骤/方法切分，而非按文章整体切分**：如「如何判断是否发生航变？」的三个排查步骤（`pdhb-c01`～`pdhb-c03`）、「如何核实客票状态？」「如何获取航变凭证？」的两种查询方法（`hskp-c01`/`hskp-c02`、`hqpz-c01`/`hqpz-c02`）各自独立成 chunk，检索命中单一方法时不会带出无关方法的内容。
6. **Metadata 字段**：每条 child 含 `level1_category`（售后票务）、`level2_category`（航班及客票状态）、`applicable_carrier`、`updated_at`（本次 5 篇源文章在导航页均无“最后更新日期”标注，全部记为 null）、`source_path`（可追溯到本地归档原文件）。
7. **无数字类结构化表格**：本子类源文章不含类似「退票」子类 131 家航司时限表那样的大规模结构化数据，因此未产出额外的 `structured_data_ref` 文件。
