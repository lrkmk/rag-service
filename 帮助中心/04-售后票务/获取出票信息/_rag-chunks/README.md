# 获取出票信息子分类 — RAG 切片

对「04-售后票务/获取出票信息」下 2 篇源文章做的切片（跳过导航列表页 `获取出票信息.md`），沿用「退票」示范的切法与 schema。

## 文件

- `parents.jsonl` — 文章级 parent 记录（2 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（7 条）

## 切分说明

1. **文章为最小一级边界**：2 篇文章各自独立产出 1 个 parent + N 个 child，互不跨界拼接。
2. **规则单元整段保留**：`如何查询出票预留信息？` 按信息类型切（出票预留邮箱 / MMB登录密码 / 支付卡号后四位 / 乘客联系方式），每类信息及其查询路径保持在同一个 chunk 内，不按字数切。`如何获取行程单？` 按获取途径切（出票预留邮箱 / 航司官网），其中“出票预留邮箱”途径下“已使用/未使用Atlas邮件服务”两种条件分支合并保留在同一 chunk（条件+结果耦合，拆开会丢失对照关系）；航司官网无法打开时的排障提示单独成 chunk（`rule_type: 例外`）。
3. **航司特例**：本分类源内容均不涉及具体航司差异，`applicable_carrier` 统一为 `通用`。仅 `cp-c02`（MMB登录密码）提到“部分航空公司（如U2）”作为举例，未构成独立航司规则，未单独打标签。
4. **Parent-Child 结构**：每个 child 都带 `parent_id`，可回带 parent 摘要用于判断规则出自「查询出票预留信息」还是「获取行程单」。
5. **Metadata 字段**：与退票示例一致，含 `level1_category`（售后票务）、`level2_category`（获取出票信息）、`applicable_carrier`、`updated_at`（两篇源文章及其父级导航文件均未标注“最后更新日期”，故记为 `null`）、`source_path`。

## 未覆盖范围

导航列表页 `获取出票信息.md` 仅罗列子文章链接，不含独立规则内容，未产出 parent/child 记录。
