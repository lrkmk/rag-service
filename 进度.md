## 1. 增量更新支持（chunk_id reconcile）

**问题**：`ingest_*.py` 原来只会 `upsert`，文档被删除或重新切片换了 `chunk_id` 后，Chroma 里的旧向量永远清不掉（孤儿数据）。

### 2.2 Markdown 标记清洗

发现嵌入 embedding 的文本里混着 `**粗体**`、`` `代码` ``、`[链接](url)`、裸露的 `#`/`##`/`###`、`>` 引用、`***` 分隔线、`&#x624D;` 这类数字 HTML 实体（部分文件如"功能和内容相关.md"重度使用，不解码会读不懂）、GitBook 的 `{% stepper %}`/`{% step %}` 组件标签、``` 代码围栏——这些都是纯噪声，会被 embedding 模型当成普通 token 计入相似度计算。

新增 `clean_markdown_text()`，在 6 个切片脚本里各自维护一份（跟这个仓库"每个脚本独立可运行、不共享 utils 模块"的既有风格一致）。过程中修了两个真实 bug：
- 单星号斜体正则会跨行误配对，把 bullet list 的 `*` 项目符号当成斜体定界符吃掉——bullet list 被压缩/损坏
- `>` 引用剥离在 `#` 标题剥离之后执行，导致 `> ### 标题` 这种嵌套在引用块里的标题没被处理干净

**状态**：✅ 已完成，全仓库残留噪声扫描已清零（agent 手写的 A类 prose 里还有少量 `` `代码` ``，不在这次脚本化清洗的范围内）。

## 4. `scripts/` 目录重构

原来 24 个文件全部平铺在 `scripts/` 下，混了切片/灌库/检索/评估/部署/开发工具六类关注点。重构为：

```
scripts/
  chunking/   切片脚本（含 chunk_diff.py）
  ingest/     灌库脚本 + parents_lookup.json
  search/     运行时查询/服务（rag_search.py、mcp_server.py 等）
  eval/       检索质量评估（含 reports/）
  deploy/     chroma_db 打包分发
  webapp/     本地切片对照工具，不进 Docker 镜像
```

`ingest_chroma.py` 改名为 `ingest_help_center.py`（原名易让人以为它管所有 Chroma 灌库，实际只灌帮助中心）。删除了两个一次性协议冒烟测试脚本 `test_mcp_remote.py`/`test_mcp_server.py`。

**踩的坑**：
- 所有脚本都用 `dirname(dirname(__file__))` 硬编码"往上两级到仓库根"，挪进子目录后全部要改成三级，逐一修了 9 个 Python 脚本 + `webapp.py`（pathlib 写法）+ 2 个部署 shell 脚本（同样的 bash 版本路径计算）。
- `.gitignore` 里 `eval/` 规则没加根锚点，新建的 `scripts/eval/` 被顺带忽略，导致 `reports/`、`eval_reply_quality.py` 会静默从 git 里消失——改成 `/eval/` 精确匹配仓库根。
- `Dockerfile` 的 `COPY`/`CMD` 同步改成新路径，保持子目录结构拷进镜像（不拍平），确保容器内目录深度和本地一致。

**状态**：✅ 已完成。全部脚本 `py_compile` 通过；9 个路径修复过的脚本逐个验证过 `REPO_ROOT` 计算正确；用现有 `chroma_db` 跑了真实查询（含跨目录 `chunk_diff`/`rag_search` 依赖）全部通过；`webapp.py` 本地起服务确认能正常出页面。**没有 Docker 环境，`docker build` 没法在这边验证，部署前建议手动跑一次。**

## 6. 生产环境实测 bug：`parents_lookup.json` 容器内过期

**症状**：服务器 `git pull` + 跑 `deploy_chroma_db.sh` 之后，查询结果里部分 chunk 的 `parent_title`/`parent_summary`/`source_path` 仍然是 `null`（如 `plsq-c02`，对应"如何批量提交退款申请？"）。本地跑同一条查询完全正常。

**根因**：`chroma_db/` 和 `scripts/ingest/parents_lookup.json` 待遇不一致。`chroma_db/` 是 gitignored 的二进制产物，通过 GitHub Release 发布/拉取，且在 `docker-compose.yml` 里做了 volume 挂载，所以 `docker compose restart` 就能生效。但 `parents_lookup.json`（虽然是 git 追踪的普通 JSON 文件，`git pull` 能更新到磁盘）之前是在 `Dockerfile` 里用 `COPY` **打进镜像**的——`docker compose restart` 复用的是旧镜像层，容器里跑的还是上次 `docker build` 那一刻打包的旧快照，`git pull` 更新的磁盘文件根本没被读取。`piliang-tijiao-tuikuan` 这条 parent 记录是在提交 `321373f` 加入的，服务器镜像如果是在这次提交之前构建的，就会复现这个 null。

**修复**：把 `parents_lookup.json` 也改成 volume 挂载（`docker-compose.yml` 新增一条 `:ro` 挂载），`Dockerfile` 不再 `COPY` 它（改成构建时写一个 `{}` 占位文件，防止脱离 docker-compose 直接 `docker run` 时因为文件不存在直接崩溃）。改完之后 `parents_lookup.json` 和 `chroma_db` 待遇完全一致：`git pull` + `docker compose restart` 就够，不需要重建镜像。

**涉及文件**：`Dockerfile`、`docker-compose.yml`、`scripts/deploy/deploy_chroma_db.sh`（更新了使用说明，明确要求先 `git pull`；顺带发现并修掉了一处指向已删除的 `test_mcp_remote.py` 的过期提示）

**状态**：✅ 代码已修复。**服务器上这次需要额外跑一次 `docker compose up --build`（完整重建）才能让这个修复本身生效**——这是最后一次需要重建；重建之后，以后的 `parents_lookup.json` 更新只需要按 `deploy_chroma_db.sh` 新流程（先 `git pull` 再跑脚本）即可，不用再重建。

## 7. `clean_markdown_text()` 转义顺序 + 嵌套引用符 bug，全语料重新切片

**问题**：审查某个已切好的 chunk（`create-order-overview`）时发现残留 `\*Dependency:\*`、孤立的 `> ` 前缀——`clean_markdown_text()`（6 个脚本各自维护一份）有两处顺序/覆盖问题：
1. 反斜杠转义还原（`\*` → `*` 等）放在函数最后一步，但粗体/斜体剥离发生在它之前——GitBook 导出里大量出现的 `\*\*Dependency:\*\*` 这种被转义的加粗标记，剥离时看到的是转义后的 `\*\*` 而不是真正的 `**`，正则匹配不上，噪声原样留在文本里。全仓库排查发现 **124 个 API 文档源文件**含这种转义标记，波及面比预想的大得多。
2. `> ` 引用符剥离只处理单层（`^>\s?`），源文档里 `> > 嵌套引用` 这种情况只剥掉第一层，第二个 `> ` 留在文本里。

**修复**：把反斜杠转义还原挪到函数最前面（先转义还原，后续正则才能认出真正的 markdown 语法），引用符剥离改成 `(?:>\s?)+` 匹配任意层数。6 个脚本 + 新增的独立小工具 `scripts/chunking/clean_markdown_text.py`（供人工切片时清洗文本片段用，不参与脚本间 import，符合"脚本独立可运行"的既有约定）全部同步。

**全语料重新生成**：
- **脚本化类型**（B/C类对比消歧+API参考、两种FAQ、产品总览、资讯）：53 次脚本调用重新跑遍所有已切过的源文档。过程中额外发现并修复了 `chunk_disambiguation.py` 的一个独立 bug——去重用的 slug 是对**调用时传入的路径字符串**（而非归一化后的 `source_path`）取哈希，两次用不同形式的路径（绝对/相对）调用同一文件会产生不同 slug，导致旧 chunk 没被正确替换、静默产生重复项（`搜索 vs 报价.md` 一度产生 3 份重复）。改成对归一化后的 `source_path` 取哈希，一次性清理所有重复项后验证 0 重复、0 残留噪声。
- **帮助中心规则单元**（59 篇文章，无脚本、纯人工切片类型）：自动清洗对已有 chunk 的 `text` 字段扫了一遍，0 处改动（说明历史内容本身不含 markdown 标记噪声）。但这只是"清洗已有文本"，不等于"重新切片"——用户明确要求彻底重新切一遍。拆成 4 组并行 subagent，每组重新读原文、按 `manual-chunking.md` 方法论重新划边界（含用新工具清洗文本），**发现的问题比预期多**：多处历史 chunk 存在改写/意译（代词替换、句式重组）、至少两处整段内容被静默丢弃（`Atlas退票服务说明.md` 缺失"航司处理时效说明"小节、`Atlas邮件服务.md` 缺失开篇的服务条款约束段落）、一处从文档其他位置错误拼接进来的句子、若干处编造的内容（不存在于原文的解释性说明）。全部修复，`chunk_diff.py` 逐篇复核，59 篇 / 290 个 chunk 全部通过 JSONL 校验 + parent/child 一致性校验 + markdown 残留噪声扫描（0 处）。

**涉及文件**：6 个 `chunk_*.py`、新增 `scripts/chunking/clean_markdown_text.py`、`scripts/chunking/chunk_disambiguation.py`（slug 归一化修复）、`doc/帮助中心/**/_rag-chunks/{children,parents}.jsonl`（19 个文件夹全部重写）、`doc/API文档/**/_rag-chunks/*.jsonl`（脚本化类型全部重新生成）。

**顺带修复**：`scripts/chunking/chunk_diff.py` 的覆盖率判定——精确匹配长度门槛从 12 降到 10；新增对"行内伪标题紧贴正文、无空行分隔"场景的兜底重试（标题行常被折进 `section` 元数据、不会重复出现在 `text` 里，之前会导致明明完整覆盖的段落被误判为"疑似改写"）。

**状态**：✅ 已完成。三个 collection（`atlas_rule_chunks` 290、`atlas_faq_chunks` 136、`atlas_api_chunks` 810、`atlas_api_faq_chunks` 124、`atlas_intro_chunks` 128、`atlas_news_chunks` 32）已全部用修复后的清洗逻辑和重新切片的内容完整重新灌库并本地验证（含对本次修复内容的真实检索验证）。
