### 快速解答

#### 我应该重试这个 Atlas API 错误吗？

仅在失败明显是暂时性的时候重试。

超时、短暂的负载高峰以及某些航司侧故障通常是可重试的。

输入错误、过期标识符、过期的附加服务和重复支付风险则不可重试。

#### 哪些错误通常需要新的标识符？

诸如 `202`、`206`、`207`、`210`、`301`、`302`、`308` 和 `631` 等代码通常意味着当前预订上下文不再有效。

从正确的更早步骤重新开始，而不是重新提交相同的标识符。

#### 何时应避免重试 `pay.do`？

当订单可能已经支付、正在出票或等待航司确认时，不要重试支付。

对于 `402`、`404`、`406` 和 `615` 等情况，先检查订单状态。

#### `getOfferPrice.do` 订单有什么变化？

在履约流程中，更积极地处理支付和出票失败。

有效窗口是 5 分钟，而不是标准的 30 分钟保留模型。

仅在确认订单仍然可支付且仍在有效窗口内后重试。

### 429 请求限制处理

Atlas 可能会以 Atlas 错误代码 `429` 拒绝超限的预订前请求。

此策略适用于以下池：

* `search.do` 使用 `10 QPS`
* `verify.do` 和 `getOffers.do` 共享 `60 QPM`
* `seatAvailability.do` 和 `getLuggage.do` 共享 `60 QPM`

`order.do` 和 `pay.do` 不在此策略范围内。

#### 响应字段

返回 `429` 时使用这些字段：

* `code` — 错误代码
* `message` — 错误摘要
* `limitType` — 活动的限制桶
* `limit` — 当前配置的限制
* `retryAfter` — 重试前的等待时间（秒）

#### 响应示例

```json
{
  "code": 429,
  "message": "Rate limit exceeded. Please retry after the current window resets",
  "limitType": "SEARCH_QPS",
  "limit": 1,
  "retryAfter": 1
}
```

使用[API 请求限制](/api-wen-dang/product-guides/booking/booking-overview/api-request-limits.md)获取完整策略、计数规则和池映射。

#### 计入限制的内容

以下请求会计入：

* 成功的请求
* 无结果的请求
* 业务失败和航司失败
* 缓存命中响应

以下请求不计入：

* 已被请求限制治理拒绝的请求
* 身份验证失败
* 参数验证失败
* 权限或安全策略拒绝
* 幂等性阻止的从未进入业务处理的重复请求

### 重试策略

在每次重试前使用此规则：

* **是**：立即或在短暂延迟后重试
* **部分**：仅在更改条件后重试
* **否**：先修复输入、流程或账户设置

典型的"部分"情况：

* `429`：等待 `retryAfter`，然后以较低频率重试
* `110`：在重试前添加回退
* `406`：等待正在进行的支付完成
* `415`：先完成 `orderCommit.do`
* `616` 或 `633`：切换支付方式
* `108`、`113`、`324`：稍后重试，不要立即重试

{% hint style="warning" %}
如果支付可能已经开始或完成，在重试前查询订单状态。
{% endhint %}

{% hint style="warning" %}
对于 `getOfferPrice.do` 订单，不要使用通用重试循环。

一旦履约窗口即将耗尽，停止重试。
{% endhint %}

### 高风险错误

以下情况导致最多的可避免的重复失败：

#### `110` 太多并发请求

不要立即重试。

添加指数退避。 从大约 `1s` 开始。 上限大约 `30s`。

#### `429` 超出请求限制

不要立即重试。

首先等待返回的 `retryAfter` 值。

然后减少突发大小、重用缓存，并减慢下一波重试。

#### `202` 路由标识符过期

不要使用相同的 `routingIdentifier` 重试 `verify.do`。

`routingIdentifier` 在 `search.do` 后最长有效 6 小时。

首先重新运行搜索。

#### `301` 会话不存在或已超时

不要重用相同的 `sessionId`。

`sessionId` 在 `verify.do` 后最长有效 2 小时。

重新运行 `verify.do` 以获取新的会话。

#### `406` 支付操作正在进行中

不要在一个支付请求仍在运行时发送另一个。

等待并查询订单状态。

#### `615` 支付完成但未能获取 PNR

**不要**重试支付。

Atlas 将手动跟进 PNR。 使用订单查询监控状态。

#### `415` 用户未确认订单

常见于 FR 流程。

在 `pay.do` 之前调用 `orderCommit.do`。

### 基于模式的检查

如果错误本身不能解释失败原因，请检查集成模式：

* 在首次 `TktInProcess` 后停止轮询
* 重用了过期的 `routingIdentifier`
* 重用了过期的 `sessionId`
* 跨会话重用了附加服务 `productCode`
* 沙箱和生产凭证混合使用
* 未配置 webhook，仅使用轮询作为唯一状态来源

当问题是由实施行为引起，而不仅仅是错误代码时，使用相关的流程页面。

### 推荐的故障排除顺序

1. 识别失败的流程
2. 检查问题是与输入、库存、标识符过期、支付还是平台相关
3. 判断代码是否可重试（是/部分/否）
4. 仅在失败是真正暂时性时重试
5. 如果相同错误重复出现，附上请求上下文升级

### 常见初步检查

* 验证凭证和账户访问
* 确认必填字段存在
* 确认标识符是新鲜的
* 在重试支付前查询订单状态
* 在重试搜索、订单或退款流程前检查余额

### 常见问题

#### Atlas API 错误代码通常帮助我决定什么？

它们帮助您判断失败是由请求输入、过期标识符、库存变化、支付状态还是暂时性平台条件引起的。

在仅依赖完整的传统表之前，请先使用上述流程特定的指南。

#### 哪些 Atlas API 错误代码通常可以安全重试？

典型的可重试情况包括短暂超时或暂时性平台条件，如 `112`、`127`、`205`、`316` 和 `705`。

始终在需要时应用退避，并避免盲目重试循环。

`429` 仅在返回的 `retryAfter` 延迟之后且以较低的请求频率重试。

当您需要区分请求限制治理和搜索并发压力时，使用 [429 vs 110](/api-wen-dang/support-and-reference/troubleshooting-and-support/errors-handing/429-vs-110.md)。

#### 哪些错误代码意味着我需要新的标识符或新的流程？

诸如 `202`、`206`、`207`、`210`、`301`、`302`、`308` 和 `631` 等代码通常意味着当前标识符、库存状态或预订上下文不再有效。

在这些情况下，从正确的更早步骤重新开始，而不是重新提交相同的请求。

#### 支付错误后我应该重试 `pay.do` 吗？

不总是。

对于 `406`，等待并先查询订单。

对于 `402`、`404` 或 `615`，在确认实际订单状态前不要重试支付。

对于 `616` 或 `633`，先更改支付条件。

对于 `getOfferPrice.do` 订单，还要确认重试仍然符合履约截止时间。
