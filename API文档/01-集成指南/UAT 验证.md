### 常见问题

#### 何时应该开始 UAT？

仅在沙箱流程端到端稳定后开始 UAT。

您的团队应能在 ATRIP 中开始验证之前，可靠地运行所需的测试场景。

#### UAT 通过证明什么？

它证明经过验证的沙箱集成已准备好进行生产切换。

它不能替代生产凭证切换或上线监控步骤。

### 本阶段的目标

验证沙箱集成已准备好进入生产环境。

此阶段将完成的沙箱集成转化为已批准的上线结果。

### 如果以下情况尚未稳定，请勿开始 UAT

当以下任何一项仍不稳定时，请等待再开始 UAT：

* 沙箱预订流程仍间歇性失败
* 所需的标识符未可靠跟踪
* 所选范围的 Webhook 处理不完整
* 您的团队无法用新的测试订单重现相同结果

### 何时开始 UAT

在以下条件满足后开始 UAT：

* 沙箱凭证已可用
* 核心预订流程已集成
* 请求头和身份验证稳定
* 如果需要完整集成 UAT，Webhook 处理已就绪
* 您的团队可以端到端执行测试用例

### ATRIP 中的 UAT 流程

在 ATRIP 的 **UAT 测试** 下完成 UAT。

使用以下路径：

1. 登录 ATRIP。
2. 打开 **UAT 测试**。
3. 选择您需要完成的功能范围。
4. 至少完成 **机票预订**。
5. 点击 **确认并继续**。
6. 填写每个测试用例所需的订单详情。
7. 点击 **提交验证**。
8. 查看自动验证结果。

如果某个用例失败，ATRIP 会直接显示失败原因。

{% hint style="info" %}
在 UAT 之前，运行 [沙箱验证测试工具包](/api-wen-dang/readme-1/sandbox-development/sandbox-validation-test-kit.md)。

在正式验证开始前，用它来确认凭证、网络访问和基本的预订路径。
{% endhint %}

### UAT 流程类型

有两种 UAT 流程。

#### 1. 搜索和出票 UAT

当您需要验证标准的搜索到出票流程时，使用此流程。

这通常包括：

* 搜索
* 校验
* 下单
* 支付
* 出票结果

#### 2. 完整集成 UAT

当您需要验证 Atlas 与您的预订平台之间的端到端集成时，使用此流程。

这通常包括：

* 乘客类型组合
* 行程类型组合
* 直飞和中转行程
* 支付方式
* 附加服务
* Webhook
* 运营跟进流程

### 如何选择正确的 UAT 流程？

对于标准的搜索到出票路径，使用 **搜索和出票 UAT**。

当您需要更广泛的覆盖范围（包括乘客组合、行程类型、附加服务、Webhook 处理和运营跟进）时，使用 **完整集成 UAT**。

### 所需的功能覆盖范围

选择与您的集成范围匹配的功能。

`机票预订` 是核心功能。

您必须在每次 UAT 运行中完成它。

如果您的上线还包括更多功能，请在同一个 UAT 范围内完成这些功能。

### 验证前应准备什么

在 ATRIP 中开始 UAT 前，准备以下项目：

* 每个测试用例的有效订单详情
* 相关的订单号或请求 ID
* 测试完整集成时的 Webhook 证据
* 您的团队使用的 Postman 集合（如果需要）

### UAT 准备检查清单

在提交验证前，确保以下项目已准备就绪：

* 确认的 UAT 范围
* 每个所需用例的新测试订单
* 可追溯的 `orderNo` 值和请求上下文
* 失败或重试用例的截图或导出的证据
* 当范围需要 Webhook 覆盖时的 Webhook 负载样本

### 最佳实践

在进入 **UAT 测试** 之前准备好订单信息。

为每个必需的用例使用新的、可追溯的测试订单。

将订单号、请求 ID、截图和 Webhook 样本与同一测试用例关联。

### 验证结果

点击 **提交验证** 后，ATRIP 会自动验证该用例。

当用例失败时，ATRIP 会直接返回错误原因。

使用该原因来更正订单数据、重新运行测试流程，或在再次提交前替换测试订单。

### 验证失败的常见原因

* `机票预订` 未完成
* 选择了错误的功能范围
* 订单详情与测试用例不匹配
* 订单状态不满足用例要求
* 完整集成 UAT 缺少所需的 Webhook 覆盖

### 如果验证失败

使用此恢复顺序：

1. 确认所选 UAT 范围与测试的工作流匹配。
2. 检查提交的订单详情是否与确切用例匹配。
3. 如果状态不明确，使用新的测试订单重新运行失败的流程。
4. 当所选范围需要时，确认 Webhook 证据存在。
5. 仅在失败条件得到纠正后再次提交验证。

### 下一步是什么？

获得批准后，继续 [生产上线](/api-wen-dang/readme-1/production-go-live.md)。

届时，申请 `LIVE` 状态、生成生产凭证、切换端点并运行受控的冒烟测试。

### 完成此阶段的标准

您已完成以下所有项目：

* 选择了正确的 UAT 流程
* 完成了所需的功能范围
* 通过了所需的验证用例
* 解决了所有报告的失败原因

### 本阶段的产出

* 通过的 UAT 结果
* 确认的上线就绪状态
* 批准的生产设置切换

### 下一步

UAT 通过后：

* 请您的客户经理将账户切换为 `LIVE`
* 生成生产凭证
* 设置生产 IP 白名单
* 替换沙箱端点
* 运行受控的冒烟测试
* 监控首批生产订单和 Webhook 事件
