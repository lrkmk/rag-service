# 429 vs 110

当需要了解 Atlas 错误代码 `429` 和 Atlas 错误代码 `110` 之间的区别时，使用本页面。

### 简要回答

`429` 是 Atlas 请求限制错误代码。

`110` 是 Atlas 业务错误，表示并发搜索请求过多。

两者都意味着请求频率过高。

它们不是同一个信号。

### 常见问题

#### 错误代码 `429` 意味着什么？

意味着当前请求速率超出了配置的请求限制策略。

在重试前使用 `retryAfter`。

#### 错误代码 `110` 意味着什么？

意味着搜索并发度高于允许的水平。

这是搜索流程中的 Atlas 业务响应。

#### 在超限流量上应该预期哪个？

当请求限制治理拒绝调用时，预期 `429`。

当搜索流程进入 Atlas 业务处理然后因并发搜索压力过高而失败时，预期 `110`。

### 核心区别

#### `429`

这是 Atlas 请求限制错误代码。

它告诉您活动的限制桶以及等待多长时间。

典型字段包括：

* `code`
* `message`
* `limitType`
* `limit`
* `retryAfter`

#### `110`

这是 Atlas 业务代码。

意味着搜索流程承受了过多的并发请求压力。

它不携带与 `429` 相同的请求限制负载。

### 如何处理每个代码

#### 对于 `429`

* 等待 `retryAfter`
* 减少突发大小
* 合并重复请求
* 避免立即并行重试

#### 对于 `110`

* 在重试前增加退避
* 降低搜索并发度
* 减少重复搜索
* 如果您的流量模式需要更高层级，请联系 Atlas

### 哪些 API 受影响？

#### `429`

Atlas 可以在请求限制治理下为选定的预订前 API 返回 `429`。

常见池包括：

* `search.do`
* `verify.do`
* `getOffers.do`
* `seatAvailability.do`
* `getLuggage.do`

#### `110`

`110` 是搜索流程问题。

将其视为搜索并发特有的问题。

### 重试指南

#### 重试 `429`

仅在返回的 `retryAfter` 延迟后重试。

然后以较低频率重试。

#### 重试 `110`

仅在增加退避并降低并发后重试。

不要在紧密循环中反复请求相同搜索。

### 快速决策规则

在生产中使用此规则：

* 如果返回的代码是 `429`，遵守 `retryAfter`
* 如果业务代码是 `110`，首先降低搜索并发

对于两种情况，改进缓存重用并抑制重复搜索突发。

### 常见错误

#### 将 `110` 视为普通超时

不要像 `112` 那样立即重试。

`110` 意味着流量压力，而不是简单的临时性超时。

#### 忽略 `429` 的 `retryAfter`

不要在返回的延迟到期前重试。

立即重试通常会导致另一个 `429`。
