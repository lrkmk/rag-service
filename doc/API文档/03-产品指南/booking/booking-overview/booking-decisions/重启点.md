当预订流程中断且您需要知道必须重新运行的最早步骤时，使用此页面。

### 简要回答

当搜索阶段上下文丢失时，从 `search.do` 重新开始。

当搜索仍然可用但预订验证必须刷新时，从 `verify.do` 重新开始。

当您在获取报价流程中且报价上下文过期时，从 `getOffers.do` 重新开始。

### 常见问题

#### 何时应从 `search.do` 重新开始？

当 `routingIdentifier` 过期或在验证或订单创建前库存发生变化时，从 `search.do` 重新开始。

#### 何时应从 `verify.do` 重新开始？

当 `sessionId` 过期或预订延迟且您需要新鲜的预订上下文时，从 `verify.do` 重新开始。

#### 何时应从 `getOffers.do` 重新开始？

当您使用获取报价流程且当前 `OfferId` 或价格上下文过期时，从 `getOffers.do` 重新开始。

### 按流程决策

#### 标准搜索流程

使用此路径：

1. `search.do`
2. `verify.do`
3. `order.do`
4. `pay.do`

从最早的中断步骤重新开始。

#### 获取报价流程

使用此路径：

1. `getOffers.do`
2. `order.do`
3. `pay.do`

当报价上下文过期时，从报价步骤重新开始。

### 在以下情况下从 `search.do` 重新开始

这些情况通常需要新的搜索：

* `202`
* `206`
* `207`
* `210`
* `302`
* 搜索阶段的行程或库存发生变化

在继续前进前使用新的 `routingIdentifier`。

### 在以下情况下从 `verify.do` 重新开始

这些情况通常需要新的验证：

* `301`
* 延迟预订但仍可用的搜索上下文
* 需要新鲜的 `sessionId`
* 需要最新的 `bookingRequirement`
* 订单创建前需要最新的附加服务上下文

使用刷新后的 `sessionId` 进行 `order.do`。

### 在以下情况下从 `getOffers.do` 重新开始

这些情况通常需要新的报价查询：

* 获取报价路径价格漂移
* 过期的 `OfferId`
* 获取报价流程中的延迟预订
* 与旧报价关联的附加服务上下文

在该流程中，使用刷新后的 `OfferId` 进行 `order.do` 和 `seatAvailability.do`。

### `308` 和价格漂移处理

`308` 表示旧的预订价格上下文不再安全。

在标准搜索流程中，从 `search.do` 重新开始。

在获取报价流程中，从 `getOffers.do` 重新开始。

### 附加服务刷新规则

如果您从 `verify.do` 或 `getOffers.do` 重新开始，也要刷新行李和座位选择。

不要将旧的附加服务 `productCode` 值带入新的预订上下文。

### 快速决策规则

在生产中使用此规则：

* 过期的 `routingIdentifier` — 从 `search.do` 重新开始
* 过期的 `sessionId` — 从 `verify.do` 重新开始
* 获取报价流程中过期的 `OfferId` — 从 `getOffers.do` 重新开始
* 过期的附加服务上下文 — 从当前验证步骤重新开始，然后刷新附加服务

### 常见错误

#### 在流程中过晚重新开始

不要这样做。

如果较早的标识符已过期，后续步骤的重试将持续失败。

#### 针对 `301` 过早重新开始

默认情况下不要这样做。

如果搜索上下文仍然可用，从 `verify.do` 再次开始。

#### 重启后保留旧的附加服务

不要这样做。

在新的验证或报价查询后刷新行李和座位上下文。
