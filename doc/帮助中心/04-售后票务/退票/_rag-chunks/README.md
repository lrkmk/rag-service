# 退票子分类 — RAG 切片示例

对「04-售后票务/退票」下 5 篇源文章做的切片示范，落地你提出的 7 条建议。

## 文件

- `parents.jsonl` — 文章级 parent 记录（5 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（23 条）
- `退票时限结构化表.json` — 从「退票申请提交时限说明」表格中抽取的 131 家航司结构化时限数据（carrier_code / limit_base / value / unit），不做语义切片，检索走精确查表

## 切分依据（对应你的 7 条建议）

1. **文章为最小一级边界**：5 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child。
2. **规则单元整段保留**：没有按字数/token 数机械切断。例如 `tps-c11`（拒单与免责说明）把"7天复核期"和"45天长期未响应"两条条件+结论耦合的规则各自保持完整一段，没有从中间切开；`fr-c01`（FR 非自愿退票判定）把"提前1小时/延后5小时"四条阈值判断作为不可再分的一个 chunk。
3. **航司特例单独打标签**：`applicable_carrier` 字段区分 `通用` / `FR` / `IJ`。FR、IJ 两篇的所有 child 都标了具体航司代码，检索时可先按 `applicable_carrier=FR` 精确匹配，未命中再 fallback 到通用退票规则，避免被通用规则稀释。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，检索命中 child 后可回带 parent 的一句话摘要，帮助模型判断这条规则出自"退票"还是易混淆的"废票/改期"（三者是同一 04-售后票务 大类下的兄弟子类，命名和场景高度相似）。
5. **FAQ 单独处理**：本次示例未覆盖 FAQ（10-常见问题），但原则一致——按一问一答切，不按字数切。如需要可以另做一版。
6. **Metadata 字段**：每条 child 含 `level1_category`（售后票务）、`level2_category`（退票）、`applicable_carrier`、`updated_at`（部分源文章带更新日期，缺失的记为 null）、`source_path`（可追溯到本地归档原文件）。
7. **数字类规则抽表**：131 家航司的退票时限没有切成 131 个 chunk，而是整体转成结构化 JSON，`tps2-c01` 这条 child 只保留判定逻辑的文字说明，并通过 `structured_data_ref` 指向结构化表——查"某航司退票要多久"应该查表，不应该做语义检索。

## 示例：一条完整 child 记录

```json
{
  "chunk_id": "fr-c01",
  "parent_id": "fr-feizyuan-tuipiao",
  "section": "2.1 非自愿退票（航班变动）",
  "rule_type": "时限规则/航司特例",
  "applicable_carrier": "FR",
  "updated_at": null,
  "text": "瑞安航空（FR）非自愿退票判定规则：航班提前1小时及以上——可退；航班提前少于1小时——不可退；航班延后5小时及以上——可退；航班延后少于5小时——不可退。最终是否符合退票条件仍需以瑞安航空的审核结果为准。"
}
```

## 未覆盖范围

这只是「退票」一个子类的示范。若切法认可，同样的规则可以套用到废票、改期、改乘客信息等其余 8 个售后子类，以及常见问题（FAQ）板块。
