# 改期子分类 — RAG 切片

对「04-售后票务/改期」下 4 篇源文章（跳过纯导航文件 `改期.md`）做的切片，沿用「退票」示例落地的 7 条切分原则。

## 文件

- `parents.jsonl` — 文章级 parent 记录（4 条），每条含一句话摘要 + child_chunk_ids
- `children.jsonl` — 规则单元级 child 记录（13 条）

## 切分依据

1. **文章为最小一级边界**：4 篇文章互不跨界拼接，每篇独立产出 1 个 parent + N 个 child（`Atlas改期政策说明`4条、`如何申请改期？`5条、`如何获取改期的报价？`3条、`改期申请的提交时限是多久？`1条）。
2. **规则单元整段保留**：`sx-c01`（改期申请提交时限）把“航班起飞前至少48小时”和“航司规定的改期提交有效期内”两个必须同时满足的条件放在同一个 chunk，没有拆开；`gq-c02`（特殊情况改期协助服务）把申请时间、处理结果、服务费用三项条件作为同一套协助服务规则整体保留。
3. **航司特例单独打标签**：`Atlas改期政策说明`中的两份航司清单（自愿改期22家、非自愿改期10家）各自单独成 chunk（`gq-c03`/`gq-c04`），`applicable_carrier` 字段直接列出该清单内全部航司代码（逗号分隔），检索"某航司能否改期"时可精确匹配，不与通用规则混淆。
4. **Parent-Child 结构**：每个 child 都有 `parent_id`，可回带 parent 摘要帮助模型区分"改期"与易混淆的兄弟子类（退票、废票等）。
5. **FAQ**：本子类源文件均为说明文/操作指南，非严格问答对，未按 FAQ 规则处理。
6. **Metadata 字段**：与退票示例字段完全一致——`level1_category`（售后票务）、`level2_category`（改期）、`applicable_carrier`、`updated_at`、`source_path`。
7. **数字类规则未抽表**：本子类没有出现类似"131家航司退票时限"那种大规模数值表，因此航司清单直接保留为文本枚举（22+10家，规模远小于退票子类那张表），未额外生成结构化 JSON 或 `structured_data_ref`。

## 本子类特有的判断取舍

- **导航文件 `改期.md` 未生成 parent/child**：与退票示例处理方式一致，它只是链接列表，无实质内容。
- **`updated_at` 全部为 null**：退票示例中的更新日期来自兄弟导航文件（`退票.md`）里每个链接后标注的"最后更新日期：xxxx年x月x日"。经核对，`改期.md` 中的四个链接均未标注更新日期，因此本子类所有 parent/child 的 `updated_at` 字段均记为 `null`（如实反映源数据缺失，未编造日期）。
- **`Atlas改期政策说明.md` 的 parent 级 `applicable_carrier`**：该文章本身是通用政策说明，但正文核心内容（二、三节）实际只对特定航司清单生效，因此没有简单标为"通用"，而是标注为"通用（含Atlas可协助改期航司清单）"，提示检索时需结合下属 child 的具体航司代码进一步判断。
- **`gq-c02` 的 `applicable_carrier`**：该条规则（48小时申请时限、不保证成功、需收服务费）文字本身未重复列出适用航司，但依据原文"适用航司清单见下文"的表述，其实际适用范围仅限于 `gq-c03`/`gq-c04` 两张清单内的航司，因此标注为"通用（仅适用于三、Atlas可协助改期航司清单内的航司）"而非直接写"通用"，避免检索时被误用到清单外航司。
- **"如何申请改期？"的注意事项拆分**：原文一个 hint 提示块包含四条不同性质的提醒（付款金额要求、不可撤回、处理时效、清单外航司建议），按内容性质拆成两个 chunk——`sq-c04`（申请流程类：付款金额+不可撤回）与 `sq-c05`（SLA类：处理时效，含清单外航司建议），而非整体保留成一个大 chunk，因为这几条提醒分属不同 `rule_type`，分开更利于精确检索。
