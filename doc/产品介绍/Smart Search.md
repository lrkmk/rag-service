# Smart Search

Smart Search 用于提升搜索覆盖率。

它适合解决缓存未覆盖、远期航班难命中和长尾航线无结果的问题。

### 这页适合谁

* 需要提升有结果率的搜索产品团队
* 需要评估长尾航线价值的运营团队
* 需要平衡覆盖率、响应速度和调用成本的技术团队

### 什么是 Smart Search

标准搜索通常优先返回缓存结果。

这种方式响应快，也更适合高频查询。

但缓存不一定覆盖所有航线和所有可售周期。

Smart Search 在必要时会补充实时搜索能力，帮助你拿到更多可售结果。

### 工作方式

当用户发起搜索时，系统会先检查缓存。

如果缓存中没有目标航线或日期，但该内容在支持航司的整体航线或 booking window 内，系统会继续向航司 API 或相关实时来源发起搜索。

这样可以在响应时间和覆盖率之间做平衡。

### 适合如何启用

#### 先用于长尾流量

先在低频航线和远期日期场景启用。

这样更容易观察覆盖率提升是否值得。

#### 再扩展到重点业务

如果转化率和覆盖率改善明显，再逐步扩大启用范围。

这样更利于控制实时调用成本。

### Smart Search 解决什么问题

* 缓存没有覆盖某条航线
* 缓存采集失败，导致无结果
* 远期出行日期超出常规命中范围
* 长尾航线搜索量低，但仍需要结果

### 适合的场景

#### 需要更高覆盖率

适合重视航线广度和可售结果率的业务。

#### 长尾航线搜索

适合搜索量不高，但不能接受空结果的场景。

#### 远期航班查询

适合需要覆盖更长 booking window 的业务。

#### 售前比价

适合在低流量场景中做实时 fare comparison。

### 不太适合的场景

#### 极致追求响应速度

如果业务更看重毫秒级稳定响应，不应把所有流量都交给 Smart Search。

#### 高频热门航线

这类场景通常更适合优先依赖缓存。

#### 对实时调用成本敏感

如果业务对额外调用成本非常敏感，应先做小范围验证。

### 使用权衡

Smart Search 的核心价值是覆盖率。

代价通常是更高的实时调用成本，或更长的响应时间。

是否启用，取决于你的业务更看重哪一项：

* 更快响应
* 更全覆盖
* 更低 L2B

### 推荐评估方式

上线前后建议持续观察这几项指标：

* 搜索有结果率
* 长尾航线覆盖率
* 平均响应时间
* L2B 或实时调用成本变化
* 启用航线的实际转化率

### 建议

* 将 Smart Search 用在长尾航线和远期日期
* 对高频热门航线，优先使用缓存结果
* 结合实际转化率评估是否扩大启用范围

### 与主搜索流程的关系

Smart Search 适合补足搜索覆盖。

它不替代下单前的验价流程。

进入预订主链路前，仍应执行标准验价与订单确认步骤。
